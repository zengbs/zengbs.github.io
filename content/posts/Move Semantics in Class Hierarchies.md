# How does compiler choose assignment operator in class hierarchies?
1. Candidates come from the LHS type (`T::operator=` and `friend` operators).
2. The RHS expression determines which overload is viable (via conversions).
3. Best match is selected using overload resolution rules.

# Avoid Defining Assignment Opeators in an Abstract Base Class

You may want to implement an abstract class as follows:
```c++
class GeoObj {
public:
    virtual ~GeoObj() = default;
    virtual void foo() = 0;
    
    // Copy assignment operator will be generated by compiler
    
    // You may want to declare move assignment
    // As declaring destructor disables move semantics
    GeoObj& operator=(GeoObj&&) = default;
protected:
    std::string name;
};

class Circle : public GeoObj
{
 ...
};
```

However, an object cannot instantiation from an abstract class. Moreover, the assignment operator in an abstract class introduces the risk of slicing issue:

```c++
Circle c1{...}, c2{...};
GeoObj& base_ref{c1};
base_ref = c2; // slicing issue
```

Thus, it's a good practice to disable assignment operators in an abstract class:
```c++
class GeoObj {
public:
    virtual ~GeoObj() = default;
    virtual void foo() = 0;
    
    // disable assignment ops to avoid slicing
    GeoObj& operator=(const GeoObj&) = delete; 
    GeoObj& operator=(GeoObj&&) = delete;
    
    // enable copy/move ctors for derived classes
    GeoObj(const GeoObj&) = default;
    GeoObj(GeoObj&&) = default;
    
protected:
    std::string name;
};
```
Here, we have to declare copy/move constructors in the base class to allow derived class to generate their respective constructors. See [here](https://hackmd.io/S1aGiGVSTqe1RMVq5Y5DWw?view#Class-Hierarchies) for details.

# Avoid Declaring Destructor in Polymorphic Derived Class
Based on the [Rules for Special Member Functions](/S1aGiGVSTqe1RMVq5Y5DWw), declaring a destructor will disable move semantics. That's why we avoid declaring destructor in a polymorphic derived class.
